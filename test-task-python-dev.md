# Тестовое задание: Python-разработчик

## О нас

Мы — торговая компания на маркетплейсах. Одна из наших внутренних задач — мониторинг позиций товаров в поисковой выдаче. Это задание проверяет навыки, которые нужны каждый день.

## Задание

### Часть 1. Парсер поисковой выдачи Ozon

Написать Python-скрипт, который:

1. Принимает **поисковый запрос** и **артикул товара (SKU)**
2. Открывает поисковую выдачу Ozon по этому запросу
3. Находит, на какой **позиции (1–100)** стоит товар с указанным артикулом
4. Если товар не найден в первых 100 результатах — возвращает `"not_found"`

**Формат вывода — JSON:**
```json
{
  "query": "нож туристический",
  "sku": "123456789",
  "position": 17,
  "page": 1,
  "total_checked": 100,
  "timestamp": "2026-03-20T14:30:00"
}
```

**Тестовые данные:** выберите самостоятельно 3 любых поисковых запроса и 3 артикула товаров на Ozon. В README укажите какие выбрали и почему.

### Часть 2. Устойчивость

- Запустите скрипт **3 раза подряд** с интервалом 30 секунд
- Все 3 запуска должны вернуть результат (не ошибку и не блокировку)
- Опишите: **что вы делаете, чтобы скрипт не заблокировали?** Какие лимиты считаете безопасными?

### Часть 3. Архитектура (текст, без кода)

Представьте, что нужно масштабировать решение:

- **500 товаров × 20 поисковых запросов = 10 000 проверок в день**
- Результаты сохраняются в PostgreSQL
- Если позиция товара упала ниже 50 — алерт в Telegram
- Сервис не должен блокироваться

**Опишите** (1–2 страницы):
- Какие технологии используете и почему
- Как обеспечите устойчивость (прокси, ротация, rate limiting)
- Примерная архитектура (можно схемой)
- Какие риски видите и как их снизите
- Примерный бюджет на инфраструктуру в месяц

## Правила

- **Язык:** Python 3.10+
- **Использовать AI — разрешено и приветствуется.** Мы не проверяем умение писать код без AI. Мы проверяем умение **решать задачи** с помощью любых инструментов.

## Что сдать

1. **Git-репозиторий** (GitHub / GitLab) с историей коммитов
2. **README.md**: как запустить, зависимости, что работает / что нет
3. **Рабочий код** (Часть 1 + 2)
4. **Архитектурное описание** (Часть 3) — отдельный файл `architecture.md`
5. **Лог подхода** — файл `approach.md`:
   - Что пробовали → почему не сработало → что сделали
   - Какие промпты давали AI и что из ответов использовали
   - Что пришлось менять руками и почему
6. **Скринкаст (запись экрана)** — 2-3 минуты:
   - Запуск скрипта с тестовым запросом
   - Видно терминал + результат в JSON
   - Повторный запуск (доказательство устойчивости из Части 2)

## Как мы оцениваем

| Критерий | Вес | Что смотрим |
|----------|-----|-------------|
| Работает | 30% | Часть 1 возвращает позицию. Часть 2 — 3 из 3 запуска успешны |
| Лог подхода | 25% | Глубина мышления, итеративность, качество работы с AI |
| Архитектура | 25% | Конкретность, реалистичность, понимание ограничений |
| Качество кода | 20% | Чистота, обработка ошибок, структура проекта |

### Что для нас важно

- **Процесс важнее результата.** Честный лог «пробовал 5 подходов, 3 не сработали, вот почему» ценнее идеального кода без объяснений.
- **Конкретика важнее общих слов.** «Прокси-пул 50 IP с ротацией каждые 10 запросов, бюджет ~$30/мес» > «нужно использовать прокси».
- **Один коммит в git = красный флаг.** Мы хотим видеть, как развивалось решение.
