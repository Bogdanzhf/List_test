# Архитектура сервиса мониторинга позиций на Ozon

## Масштаб задачи

500 товаров × 20 поисковых запросов = **10 000 проверок в день**.
При равномерном распределении: ~7 проверок в минуту, ~416 в час.

---

## Стек технологий

### Очередь задач — Celery + Redis

Celery управляет распределением задач между воркерами.
Redis — брокер очереди и хранилище промежуточных результатов.

Почему именно они:
- Celery поддерживает rate limiting на уровне задачи:
  `@app.task(rate_limit="10/m")` — не более 10 запросов в минуту
- Redis даёт sub-millisecond latency для очереди
- Стек знакомый, проверенный в продакшене

### База данных — PostgreSQL

Схема таблиц:

```sql
-- Товары и запросы
CREATE TABLE checks (
    id SERIAL PRIMARY KEY,
    sku VARCHAR(20) NOT NULL,
    query VARCHAR(255) NOT NULL,
    position INTEGER,           -- NULL если not_found
    page INTEGER,
    total_checked INTEGER,
    checked_at TIMESTAMP DEFAULT NOW(),
    proxy_used VARCHAR(50)      -- для аудита
);

-- Индексы для аналитики
CREATE INDEX idx_checks_sku ON checks(sku);
CREATE INDEX idx_checks_checked_at ON checks(checked_at);
```

Почему PostgreSQL, а не MongoDB: данные строго структурированы,
нужны аналитические запросы типа "позиция товара за последние 7 дней".

### Уведомления — Telegram Bot API

При падении позиции ниже 50 Celery-воркер отправляет POST-запрос
к Telegram Bot API напрямую (без дополнительных библиотек).

Условие срабатывания алерта:
```python
if position != "not_found" and position > 50:
    send_telegram_alert(sku, query, position)
```

│  #1   │ │  #2   │     каждый воркер использует свой прокси

    │         │
    ▼         ▼

         ▼

## Устойчивость к блокировкам

### Прокси-пул

Тип: **резидентные прокси** (residential), не датацентровые.
Датацентровые IP Ozon блокирует агрессивно — они в чёрных списках.

Провайдеры: Bright Data, Oxylabs, Smartproxy.
Минимальный пул: **50 IP**, ротация каждые 10 запросов.

### Rate limiting
10 000 проверок / день = 416 / час = ~7 / минуту

Безопасный лимит с одного IP: **1 запрос в 3-5 минут**.
При пуле 50 IP: 50 × (1 запрос / 4 мин) = 12-13 запросов в минуту.
Это перекрывает потребность с запасом.

### Ротация User-Agent и fingerprint

- Пул из 20+ реальных User-Agent строк Chrome (Windows/Mac)
- Случайное viewport при каждом запросе
- Playwright stealth патчит navigator.webdriver и другие JS-сигнатуры

### Retry-стратегия
При блокировке: пауза 5 мин → смена прокси → повтор
Максимум 3 попытки на задачу
После 3 неудач: задача в dead letter queue, алерт разработчику

## Риски и их снижение

| Риск | Вероятность | Решение |
|---|---|---|
| Ozon меняет HTML-структуру | Высокая | Мониторинг ошибок, быстрый хотфикс селекторов |
| Прокси попадают в бан | Средняя | Ротация пула, метрики по ban rate каждого IP |
| Резкий рост числа товаров | Низкая | Горизонтальное масштабирование воркеров |
| Капча на всех IP | Низкая | Пауза 24ч + смена всего пула |
